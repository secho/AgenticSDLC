The AI SDLC platform landscape

Every major dev-tools vendor has shipped some version of "agents that write and ship code" in the last eighteen months. Before you architect your own agentic SDLC, it helps to know what the market already sells — where it's strong, where it locks you in, and where the gaps are that a vendor-agnostic architecture fills.

Platform Profiles

Three groups, doing different jobs. Governance layers manage agents they don't provide. Autonomous coding agents do the work. IDE-native and ecosystem assistants keep a human in the loop. Most real stacks end up combining one from each.

Governance & orchestration

These don't write code themselves — they sit above the agents that do, adding the context, policy, identity, and cost control an organization needs before it lets agents loose at scale.

JetBrains Central

Governance layer · Model-agnostic

Not an IDE or a model — an orchestration and governance layer that sits above whichever agents a team already runs (Claude Agent, Codex, Gemini CLI, or custom). It adds policy enforcement, identity, cost attribution, and cloud execution infrastructure across the tools a team already uses, and lets agents collaborate with humans through Slack, Atlassian, and Linear. JetBrains frames it around a survey finding it published in early 2026: ~90% of developers use AI, but only ~22% use coding agents — the bottleneck has moved from generating code to governing it. It manages agents; it doesn't replace the need to select and pay for them separately.

  • PricingShifting to on-demand AI credits; Early Access Program in Q2 2026
  • BackingFrom JetBrains (IntelliJ, PyCharm); launch partners Google Cloud, Anthropic, OpenAI
  • Best forGoverning policy and cost across many agents, models, and clouds
  • Watch forEarly — limited rollout through 2026, not yet generally available

Port.io

Agentic engineering platform (IDP) · Model-agnostic

An internal developer portal that repositioned in 2026 as an agentic engineering platform. Rather than ship its own coding agent, it becomes the control plane: a Context Lake (a pre-joined catalog of services, ownership, and standards) that agents query, workflow orchestration, an AI Registry cataloguing every agent, skill, and MCP server in the org, and governance that scopes what each agent can read and do via RBAC. Claude, Cursor, and Copilot reach it through Port's MCP server. Its pitch is taming "agentic chaos" — and cost: Port reports its Context Lake cut per-query token spend 56–80% versus agents hitting raw MCP connections, by not re-deriving system relationships on every run.

  • PricingFree tier; Basic $30 and Standard $40 per seat/mo; custom Enterprise
  • Backing~$158M raised, ~$800M valuation (Series C led by General Atlantic, late 2025)
  • Best forPlatform teams unifying agent context, catalog, and governance on an IDP
  • Watch forA control plane, not a coding agent — you still bring the agents

Autonomous coding agents

These do the actual work — plan a task, execute it in a sandbox, and hand back a pull request. They range from fully unattended to human-supervised.

Factory.ai

Multi-surface Droids · Model-agnostic

Organizes agents by job function rather than shipping one general-purpose assistant — a Code Droid ships changes, a Reliability Droid investigates incidents, a Knowledge Droid writes docs, a Product Droid drafts PRDs. They run from a terminal, desktop app, SDK, or headless in CI/CD, and work against any leading frontier or open-weight model. Factory commits to not training on customer code and operates under SOC 2 Type II. Its broad "run agents across the whole delivery workflow" framing is the ambition; the caveat is cost predictability under heavy use.

  • PricingPro $20, Plus $100, Max $200 per month; Enterprise on request
  • Traction~$1.5B valuation; used at Nvidia, Adobe, Morgan Stanley, MongoDB; ranked #1 on Terminal-Bench
  • Best forTeams wanting agents across the full delivery workflow, not just editing
  • Watch forUsage costs can balloon on heavy autonomous workloads

Devin (Cognition)

Autonomous coding agent · Cognition

Popularized the "assign it a ticket" model: it plans, executes inside an isolated VM with its own browser, terminal, and editor, and opens a PR with no supervision during the run — several tickets in parallel, each in its own sandbox. Pricing moved from a flat $500/seat to usage-based billing (ACUs, a normalized measure of VM time and inference), which drove rapid growth. Cognition also acquired the Windsurf IDE. Independent testing consistently puts real-world success rates well below its benchmark headline, so budget review time on anything non-trivial.

  • PricingUsage-based (ACUs); tiers from free up to ~$500/mo Team, plus Enterprise
  • TractionCognition valued ~$26B; Nubank ran a 6M-line migration on it, citing large efficiency gains
  • Best forParallel, well-scoped ticket backlogs and large-scale migrations
  • Watch forReal-world success trails the benchmark headline — review overhead is real

GitHub Copilot — Agent Mode & Coding Agent

In-editor + async cloud agent · Multi-model

Two tiers span the workflow. Agent Mode makes supervised multi-file edits inside the IDE (VS Code, Visual Studio, JetBrains, Eclipse, Xcode); the separate Coding Agent takes a GitHub Issue and returns a PR asynchronously from a GitHub Actions sandbox. Every PR the Coding Agent opens triggers CodeQL, secret scanning, and Dependabot automatically, and MCP lets it pull context from databases, Confluence, or Figma mid-task. The tradeoff is ecosystem gravity — it assumes GitHub as the system of record — and a known gap: .copilotignore exclusions don't apply to the Coding Agent.

  • PricingFree; Pro $10, Pro+ $39, Business $19, Enterprise $39; usage-based for agentic features
  • Traction~20M users, ~4.7M paid; the coding agent is used across ~90% of the Fortune 100
  • Best forTeams standardized on GitHub as their system of record
  • Watch forEcosystem lock-in; ignore-file exclusions don't bind the Coding Agent

Anthropic Claude Code

Terminal/IDE agent + enterprise platform · Claude only

Started as a terminal coding assistant and grew an enterprise layer on top: Dynamic Workflows fan a single request out to dozens or hundreds of parallel sub-agents for codebase-scale refactors and migrations; Managed Agents provide hosted infrastructure for stateful, long-running agents so teams don't build the execution loop themselves; an open Skills standard and plugin marketplace let orgs package and deploy reusable capabilities; and admin controls add spend caps, analytics, and model routing — much of what JetBrains Central and Port aim to standardize across vendors, but for Claude specifically. Deployable on Anthropic's cloud, Azure, or self-hosted, with a US data zone for residency.

  • PricingVia Claude Pro/Max subscriptions and API; enterprise spend caps and analytics
  • StrengthsParallel sub-agents, Managed Agents, Skills standard, plugin marketplace
  • Best forDeep interactive engineering and codebase-scale refactors/migrations
  • Watch forClaude-only — no choice of underlying model

Google Jules

Async coding agent · Gemini only

Built for fire-and-forget review: no editor, no local mode — everything clones into a Google Cloud VM, where Jules drafts a plan, executes, adds tests alongside the implementation, and opens a PR. Strong on scoped, well-defined tasks; weaker on architectural judgment and large-codebase context, matching its reputation as a capable junior developer that still needs oversight. Teams often pair it with an interactive tool — Jules for long-running or overnight work, something like Claude Code or Cursor for the in-the-loop engineering.

  • PricingFree (15 tasks/day); AI Pro $19.99 and AI Ultra $124.99 per month by task quota
  • Runs onGemini, inside a Google Cloud VM — no local or editor surface
  • Best forFire-and-forget PRs on routine, well-scoped tasks
  • Watch forWeak on complex/architectural work; execution is Google-Cloud-only

OpenAI Codex

CLI + cloud coding agent · GPT-5.x-Codex

Covers the same asynchronous issue-to-PR pattern as Jules, but spans the terminal, the ChatGPT app, and the cloud, routing over coding-specific GPT-5.x variants. Its differentiator is an open-source CLI (Apache-2.0) that teams can audit and run locally — useful where the cloud agents' code upload or Jules' Google-VM-only model is a non-starter on data residency. Switch to the local CLI when you need strict control; use the cloud when you want fire-and-forget.

  • PricingBundled with paid ChatGPT plans; API-rate billing for overflow usage
  • SurfacesOpen-source CLI (Apache-2.0), ChatGPT app, and OpenAI cloud
  • Best forTeams wanting an auditable, self-hostable CLI alongside cloud PRs
  • Watch forCloud agents upload code to OpenAI — use the CLI for strict control

IDE-native & ecosystem-specific assistants

These keep a developer in the driver's seat — agentic help delivered inside an editor or tied to a specific ecosystem, rather than run unattended.

Cursor

AI-native IDE · Multiple models

An IDE first — a VS Code fork with agentic editing built in rather than bolted on: Composer for multi-file changes, Rules/Commands for repeatable behavior, and background cloud agents for parallel work. It embeds code chunks for retrieval (a RAG approach to codebase context) and lets developers pick the underlying model. The fastest path for someone who wants high-autonomy editing without leaving a familiar editor shell, but tuned for solo and small-team speed rather than org-wide governance.

  • PricingSubscription IDE with per-tier request limits; multiple selectable models
  • StrengthsComposer multi-file edits, background cloud agents, multi-agent workflows
  • Best forFast, high-autonomy in-editor work for individuals and small teams
  • Watch forOptimized for editing speed over organization-wide governance

Amazon Q Developer

Cloud-native assistant · Amazon Bedrock

An AWS modernization accelerator more than a general coding agent — built on Amazon Bedrock, IAM-aware, and integrated across IDEs and the AWS Console with chat in Slack and Teams. It's strongest on tasks tied to AWS infrastructure: migrations, modernization, and console automation. Analysts frame it as complementary to editor-native tools rather than a replacement — the AWS operations layer alongside your IDE. Off an AWS-centric stack, the model-agnostic options on this list are a stronger general-purpose choice.

  • PricingFree tier plus paid Pro; part of the AWS ecosystem
  • Runs onAmazon Bedrock, IAM-aware, across major IDEs and the AWS Console
  • Best forAWS-native modernization, migrations, and console-driven tasks
  • Watch forWeaker general-purpose choice off an AWS-centric stack

Sourcegraph Cody & Amp

Search-backed assistant + agent · Enterprise LLMs / model-agnostic

Sourcegraph split its offering in two. Cody leans on Sourcegraph's code-search graph — a structural advantage in sprawling, multi-repo estates — but went enterprise-only in 2025 (its free and pro tiers were retired), lags newer tools on raw agentic capability, and can be slow on multi-million-line monorepos. Amp is the separate, more agentic product: model-agnostic routing, multi-agent orchestration, shared team threads, and zero-data-retention deployment for compliance-sensitive teams. Many orgs keep Sourcegraph for search and pair it with a separate agent for execution.

  • PricingCody enterprise-only (free/pro retired 2025); Amp sold separately
  • StrengthsCody = code-search-graph context; Amp = model-agnostic multi-agent + ZDR
  • Best forMassive multi-repo estates where finding code is the real bottleneck
  • Watch forCody lags on agentic depth; latency on very large monorepos
The Takeaway

Almost every coding agent above bundles the same three layers together: which agents you can use, how they're governed, and which cloud they run on. Pick one and you've usually picked all three. The governance layers — JetBrains Central and Port.io — are the clearest sign the market is splitting those apart: control planes that manage whichever agents, models, and clouds you choose. That's exactly the architecture this site argues for: agents, governance, and cloud infrastructure as three independently replaceable layers, not a single vendor's bundle.

That's the difference between adopting a platform and building the capability yourself. The right platforms above can still be part of your stack — the question is whether your architecture lets you swap any one of them later without rebuilding the rest.
